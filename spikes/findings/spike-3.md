# Spike 3: Structured Data Extraction

## Result: PASS

We can get structured key-value output from a workflow run, natively.

## Three Ways to Get Results

### 1. `workflow.run()` — Raw step results
```python
result = await workflow.run()
# result.step_results: List[ActionResult]
# Each ActionResult has:
#   .extracted_content: str  (free text from LLM extraction)
#   .success: bool | None
#   .error: str | None
```
The `extract_page_content` step returns free-text in `extracted_content`. It's a string, not structured JSON — the LLM extracts whatever the `goal` field asks for.

### 2. `workflow.run(output_model=PydanticModel)` — Structured output (BEST)
```python
from pydantic import BaseModel, Field

class ProductData(BaseModel):
    title: str = Field(description="Product title")
    price: str = Field(description="Listed price")
    in_stock: bool = Field(description="Whether in stock")

result = await workflow.run(output_model=ProductData)
print(result.output_model)
# ProductData(title='Apple AirPods Max', price='$549.00', in_stock=True)
```

This is the ideal approach for Construct:
- Pass a Pydantic model as `output_model`
- After all steps complete, workflow-use feeds the accumulated extraction results to the LLM with the model schema
- Returns a fully typed, validated Pydantic instance
- Works with the extraction step's `goal` field — the LLM maps free-text to structured fields

### 3. `workflow.run_as_tool(prompt)` — JSON string for MCP
```python
result_json = await workflow.run_as_tool("Extract product data")
# Returns: '{"success": true, "steps_executed": 2, "inputs_used": {}, "context": {}}'
```
- Parses inputs from natural language prompt
- Returns JSON string with metadata
- Context dict holds extracted variables
- Good for MCP tool interface, but less structured than output_model

## Extraction Step Types

Two extraction step types exist:

| Type | Field | Usage |
|------|-------|-------|
| `extract_page_content` | `goal: str` | Generated by deterministic mode. LLM extracts from page. |
| `extract` | `extractionGoal: str` | Generated by LLM mode. Same behavior. |

Both use the `page_extraction_llm` to analyze page content and return text.

## Extraction Quality

With gpt-4o-mini, extraction is accurate for simple pages:
- Page title: exact match
- Paragraph text: exact match
- Link text: correct ("Learn more")

The `goal` field is a natural language instruction, so it can be as specific or broad as needed.

## Construct Integration Plan

1. User defines output schema during recording (field names, types, descriptions)
2. We convert this to a Pydantic model at runtime
3. Workflow generation adds an `extract_page_content` step with a goal derived from the schema
4. At replay time, pass the Pydantic model as `output_model` to `workflow.run()`
5. Return the typed result to the frontend / MCP

No bolt-on extraction needed — the native `output_model` support does exactly what we need.
